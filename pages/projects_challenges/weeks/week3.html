CSS Flexbox and Alignment

In Week 3, we continued our journey with CSS, focusing on mastering the Flexbox layout model and understanding the nuances of CSS alignment properties. Here are the key topics we covered:

  • Flexbox: Flexbox, or the Flexible Box Layout, is a one-dimensional layout method for laying out items in rows or columns. We learned how to create flexible and responsive web layouts using the flex container and flex items.
  • Flex Container: Discussed the properties of the flex container such as display: flex, flex-direction, flex-wrap, and justify-content which define how flex items are laid out in the flex container.
  • Flex Items: Explored flex item properties including flex-grow, flex-shrink, and flex-basis to control the sizing and ordering of items within a flex container.
  • CSS Alignment: Learned about various CSS properties used for aligning elements, such as align-items, align-self, and align-content, crucial for precise positioning within flex containers.
  • Responsive Design: Applied Flexbox in practical scenarios to create responsive web designs that adapt to different screen sizes, enhancing the user experience.
  • Real-world Projects: Implemented Flexbox in several mini-projects, understanding its importance in modern web development and its role in creating fluid, scalable layouts.

The knowledge gained this week has been pivotal in understanding the layout challenges and how Flexbox provides elegant solutions, making web design more efficient and visually appealing

Relational Databases

Data inconsistency

Where the same data exists in more than one place, and the data is not in the same state, i.e. different data formats, or has been changed sol they no longer match.


Data redundancy

Where the same data exists in more than one

foreign key

A foreign key is a field/column (or combination of field/columns) that is used to establish and enforce a link between the data in two tables to control the data that can be stored in the foreign key table.

Advantages

Relational databases are easier to manage and maintain. They are also more flexible and scalable.

Flexibility. It's easy to add, update, or delete tables, relationships, and make other changes to data whenever you need without changing the overall database structure or impacting existing applications.

Accurate. You are less likely to have redundant or inconsistent data. Security. Because data is broken into entities, you can make some entities read only, or even deny access to some users.

types of relationship databases

  • one-to-one
  • one-to-zero or zero-to-one
  • one-to-many
  • many-to-one
  • many-to-many
one-to-one

One to One Tables are related in such a way that one record in a table relates to only one record in a second table.

Tables are related in such a way that one record in a table relates to only one or more records in another table.

many-to-many

Tables are related in such a way that multiple records in a table relate to multiple records in another table.

problems with many to many

Many-to-many relationships can be difficult to manage because the database can't directly reference the relationship. Instead, the database must use an intermediary table to reference the two other tables. This intermediary table is called a junction table. The junction table contains the primary keys of both tables and creates a many-to-one relationship between the junction table and each of the other two tables.

Normalisation

A process to help you decide how and where to split a database into more tables and why.

For a table to be in the first normal form, it must meet the following criteria: • a single cell must not hold more than one value atomicity) • there must be a primary key for identification • There must be no duplicated rows or columns • each column must have only one value for each row in the table • Each row must be unique

Second Normal Form - 2NF
The 1NF only eliminates repeating groups, not redundancy. That's why there is 2NF. A table is said to be in 2NF if it meets the following criteria: • It's already in 1NF • It has no partial dependency. That is, all non-key attributes are fully dependent on a primary key.

Third Normal Form - 3NF

Third normal form is the final stage of the most common normalization process. The rule for this is: Fulfils the requirements of 2NF Has no transitive functional dependency What does this even mean? What is a transitive functional dependency? It means that every attribute that is not the primary key must depend on the primary key and the primary key comanly.
